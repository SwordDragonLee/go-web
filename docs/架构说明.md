# Gin 分层架构说明

## 项目分层架构概览

本项目采用经典的**三层架构**（Handler → Service → Repository），结合 Gin 框架实现 RESTful API。

```
┌─────────────────────────────────────────────────────────┐
│                    HTTP 请求层                           │
│  (Gin Router + Middleware)                              │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│                  Handler 层 (控制器层)                    │
│  - 接收 HTTP 请求                                        │
│  - 参数验证和绑定                                         │
│  - 调用 Service 层                                       │
│  - 返回 HTTP 响应                                        │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│                  Service 层 (业务逻辑层)                  │
│  - 业务逻辑处理                                           │
│  - 数据验证和转换                                         │
│  - 调用 Repository 层                                    │
│  - 事务管理（如需要）                                     │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│              Repository 层 (数据访问层)                   │
│  - 数据库 CRUD 操作                                       │
│  - SQL 查询                                              │
│  - 数据模型映射                                           │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│                    Database                              │
│  (MySQL via GORM)                                        │
└─────────────────────────────────────────────────────────┘
```

## 详细分层说明

### 1. Router 层（路由层）

**位置**: `internal/api/router/router.go`

**职责**:

- 定义 API 路由和端点
- 注册中间件（认证、权限等）
- 将请求路由到对应的 Handler

**示例代码**:

```go
func SetupRouter(userHandler *handler.UserHandler) *gin.Engine {
    r := gin.Default()

    v1 := r.Group("/api/v1")
    {
        auth := v1.Group("/auth")
        {
            auth.POST("/register", userHandler.Register)
            auth.POST("/login", userHandler.Login)
        }

        user := v1.Group("/user")
        user.Use(middleware.AuthMiddleware())  // 认证中间件
        {
            user.GET("/profile", userHandler.GetProfile)
        }
    }

    return r
}
```

**特点**:

- 使用 Gin 的路由组（Group）组织 API
- 中间件链式调用
- RESTful 风格的路由设计

---

### 2. Middleware 层（中间件层）

**位置**: `internal/api/middleware/`

**职责**:

- 请求预处理（认证、授权、日志等）
- 统一错误处理
- 跨域处理（CORS）
- 请求限流

**示例代码**:

```go
// JWT 认证中间件

    return func(c *gin.Context) {

```

**中间件执行顺序**:

```
请求 → Middleware1 → Middleware2 → Handler → Middleware2 → Middleware1 → 响应
```

---

### 3. Handler 层（控制器层）

**位置**: `internal/api/handler/`

**职责**:

- 接收 HTTP 请求
- 参数绑定和验证（使用 Gin 的 `ShouldBindJSON` 等）
- 调用 Service 层处理业务逻辑
- 格式化 HTTP 响应
- 错误处理和状态码设置

**示例代码**:

```go
func (h *UserHandler) Register(c *gin.Context) {
    // 1. 参数绑定
    var req dto.RegisterRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(http.StatusBadRequest, dto.Error(...))
        return
    }

    // 2. 调用 Service 层
    userInfo, err := h.userService.Register(&req)
    if err != nil {
        c.JSON(http.StatusBadRequest, dto.Error(...))
        return
    }

    // 3. 返回响应
    c.JSON(http.StatusOK, dto.Success(userInfo))
}
```

**特点**:

- 薄控制器（Thin Controller）：只负责请求/响应处理
- 不包含业务逻辑
- 统一的响应格式

---

### 4. Service 层（业务逻辑层）

**位置**: `internal/service/`

**职责**:

- 实现业务逻辑
- 数据验证和业务规则检查
- 调用多个 Repository 组合数据
- 事务管理
- 调用外部服务（如邮件服务）

**示例代码**:

```go
func (s *userService) Register(req *dto.RegisterRequest) (*dto.UserInfo, error) {
    // 1. 业务验证
    if _, err := s.userRepo.GetByUsername(req.Username); err == nil {
        return nil, errors.New("用户已存在")
    }

    // 2. 数据处理
    hashedPassword, _ := utils.HashPassword(req.Password)

    // 3. 创建用户
    user := &model.User{
        Username: req.Username,
        Password: hashedPassword,
        // ...
    }

    // 4. 调用 Repository
    if err := s.userRepo.Create(user); err != nil {
        return nil, err
    }

    return s.toUserInfo(user), nil
}
```

**特点**:

- 接口定义，便于测试和替换实现
- 业务逻辑集中管理
- 可组合多个 Repository

---

### 5. Repository 层（数据访问层）

**位置**: `internal/repository/`

**职责**:

- 数据库 CRUD 操作
- SQL 查询构建
- 数据模型与数据库表的映射
- 数据库连接管理

**示例代码**:

```go
func (r *userRepository) GetByID(id uint) (*model.User, error) {
    var user model.User
    err := r.db.Where("id = ?", id).First(&user).Error
    if err != nil {
        return nil, err
    }
    return &user, nil
}

func (r *userRepository) List(page, pageSize int, keyword string, ...) ([]*model.User, int64, error) {
```

**特点**:

- 使用 GORM 简化数据库操作
- 接口定义，便于切换数据库
- 专注于数据访问，不包含业务逻辑

---

### 6. Domain 层（领域模型层）

**位置**: `internal/domain/`

**职责**:

- 定义数据模型（Model）
- 定义数据传输对象（DTO）
- 定义常量（Constants）
- 领域规则和验证

**结构**:

```
domain/
├── model/          # 数据模型（对应数据库表）
├── dto/            # 数据传输对象（API 请求/响应）
└── constant/       # 常量定义
```

**示例**:

```go
// Model: 数据库模型
type User struct {
    ID       uint   `gorm:"primarykey"`
    Username string `gorm:"uniqueIndex"`
    Password string
    // ...
}

// DTO: API 请求对象
type RegisterRequest struct {
    Username string `json:"username" binding:"required"`
    Password string `json:"password" binding:"required,min=6"`
    // ...
}
```

## 数据流向

### 请求流程

```
1. HTTP 请求
   ↓
2. Gin Router (路由匹配)
   ↓
3. Middleware (认证、日志等)
   ↓
4. Handler (参数绑定、验证)
   ↓
5. Service (业务逻辑处理)
   ↓
6. Repository (数据库操作)
   ↓
7. Database (MySQL)
```

### 响应流程

```
1. Database 返回数据
   ↓
2. Repository 返回 Model
   ↓
3. Service 转换为 DTO
   ↓
4. Handler 格式化为 JSON
   ↓
5. Middleware (可选处理)
   ↓
6. HTTP 响应返回客户端
```

---

## 依赖关系

```
Router → Handler → Service → Repository → Database
         ↓         ↓          ↓
       DTO      Model      GORM
```

**依赖规则**:

- ✅ Handler 依赖 Service
- ✅ Service 依赖 Repository
- ✅ Repository 依赖 Model
- ❌ Handler 不直接依赖 Repository
- ❌ Service 不直接操作数据库

---

## 优势

### 1. **关注点分离**

- 每一层职责明确，易于维护

### 2. **可测试性**

- 接口定义便于 Mock 测试
- 各层可独立测试

### 3. **可扩展性**

- 易于添加新功能
- 易于替换实现（如切换数据库）

### 4. **代码复用**

- Service 层可被多个 Handler 调用
- Repository 层可被多个 Service 调用

### 5. **团队协作**

- 不同层可由不同开发者负责
- 接口定义作为契约

---

## 最佳实践

### 1. Handler 层

- ✅ 保持简洁，只处理 HTTP 相关逻辑
- ✅ 统一错误处理和响应格式
- ❌ 不要包含业务逻辑

### 2. Service 层

- ✅ 实现所有业务逻辑
- ✅ 使用接口定义，便于测试
- ✅ 处理事务（如需要）
- ❌ 不要直接操作数据库

### 3. Repository 层

- ✅ 只负责数据访问
- ✅ 使用接口定义
- ❌ 不要包含业务逻辑

### 4. 错误处理

- Handler 层：处理 HTTP 错误（400, 401, 500 等）
- Service 层：返回业务错误
- Repository 层：返回数据库错误

## 总结

本项目采用**经典三层架构**，结合 Gin 框架实现：

1. **Router 层**：定义路由和中间件
2. **Handler 层**：处理 HTTP 请求/响应
3. **Service 层**：实现业务逻辑
4. **Repository 层**：数据访问
5. **Domain 层**：领域模型和 DTO

这种架构清晰、可维护、可测试，是企业级 Go Web 应用的标准架构模式。
